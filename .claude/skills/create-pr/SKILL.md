# create-pr skill

Produce a **consistently structured** pull request for the current branch. The
template below is the single source of truth: every PR this repo opens MUST use
it, in this section order, so that (a) an AI agent can read one PR and fully
understand what the branch does and why, and (b) any change can be reverted
cleanly later by following the explicit revert map.

Optional `$ARGUMENTS`: free text used as a title hint and/or extra context
(e.g. `/create-pr split stream reliability fixes; draft`). Honor `draft` if present.

---

## Step 1 — Preconditions & context gathering

Run these read-only checks first and keep the outputs; you will quote real
values into the template (never invent SHAs, file names, or test counts).

```bash
# Repo + branch
git rev-parse --show-toplevel
git branch --show-current
git remote -v

# Base branch: default to `main` unless the user named another in $ARGUMENTS
git log --oneline origin/main..HEAD            # commits unique to this branch
git diff --stat origin/main...HEAD             # file-level change map
git status --porcelain                         # uncommitted work?
```

Rules:
- **Base branch** is `main` unless the user specified otherwise.
- If `git status` shows **uncommitted changes**, STOP and tell the user: commit
  first (offer to, or suggest `/speckit-git-commit`). A PR must reflect committed
  state — never open a PR with a dirty tree silently.
- If the current branch **is** `main`, STOP and ask which feature branch to use
  (do not open a PR from `main`).
- Identify the **Spec Kit feature** if any: look for `specs/NNN-*/` touched by the
  diff and read its `spec.md`/`plan.md` summary lines to cite them.
- Confirm `gh` works (`gh repo view --json nameWithOwner -q .nameWithOwner`). If
  the remote is SSH and push fails, the repo's HTTPS remote + token must be used
  (see project CLAUDE.md "Git Authentication").

## Step 2 — Decompose the branch into LOGICAL CHANGE UNITS

This is the most important step for revertability. Group the commits/files into
**logical units** — each a self-contained concern that could be reverted on its
own (e.g. "pre-open warmup feature", "stream fatal-error handling",
"stale-data freshness fix"). For each unit capture:
- a short name,
- the commits (SHAs) and key files that compose it,
- one line on why it exists,
- whether it is independently revertable and any ordering dependency.

A branch with one concern has exactly one unit. Do not merge unrelated concerns
into one unit just because they share a branch.

`None.` (never delete a section — uniform structure is the point). Keep prose
tight; prefer tables and bullets. Quote real paths, SHAs, and numbers.

## Step 4 — Title

Format: `<type>(<scope>): <imperative summary>` (Conventional-Commits style),
matching this repo's history (e.g. `feat(023): pre-open warmup for live paper trading`,
`fix(024): stop hammering Alpaca on fatal websocket errors`). Use the Spec Kit
feature number as scope when there is one. Prefer the user's title hint from
`$ARGUMENTS` if given.

## Step 5 — Push and open the PR

```bash
# Push the current branch to the GitHub remote (HTTPS; token-authed per CLAUDE.md)
git push -u origin HEAD

# Write the filled body to a temp file to preserve formatting, then open the PR.
gh pr create --base <base-branch> --head "$(git branch --show-current)" \
  --title "<title>" --body-file <path-to-body> [--draft]
```

- Use `--body-file` (not inline `--body`) so markdown/code fences survive.
- Add `--draft` if the user asked for a draft.
- If `gh pr create` reports a PR already exists for the branch, switch to
  `gh pr edit <num> --title ... --body-file ...` to update it in place instead.
- After creation, print the PR URL returned by `gh`.

## Step 6 — Report

Tell the user: the PR URL, the title, and a one-line list of the logical units.
Do NOT merge. Do NOT push to `main`.

---

## Notes & guardrails
- **Never fabricate** SHAs, file paths, test counts, or spec ids — derive every
  value from the Step 1/2 commands.
- **One concern per logical unit:** if the branch mixes a feature with unrelated
  reliability fixes, list them as separate units (and mention they could have been
  separate PRs) so a future revert can surgically target one.
- **Commit-message trailers** (separate from this PR body) end with:
  `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`.
- Keep the PR body the single artifact a future agent needs: someone reading only
  this PR should understand the branch and know exactly how to undo it.
